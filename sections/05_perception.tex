\subsubsection*{Lane Segmentation Architecture}
\begin{itemize}
    \item \textbf{ZED2 Hardware and Spatial Sensor Fusion:} The pipeline begins at the hardware level with the ZED2 stereo camera, which serves as our primary optical and spatial sensor. The camera features a built-in Inertial Measurement Unit (IMU), which the onboard SDK utilizes to continuously calculate its own internal Extended Kalman Filter (EKF). This hardware-level sensor fusion delivers a highly stable and drift-corrected positional pose, ensuring that our generated 3D point cloud remains consistent in physical space regardless of vehicle vibrations or rapid frame-to-frame displacements. 
    
    \item \textbf{Traditional CV vs. Deep Learning:} Initially, we attempted a classical computer vision approach using Canny Edge Detection coupled with a Hough Transform. While this method was computationally inexpensive, it proved extremely brittle in unpredictable outdoor environments. Shadows, sun glare, and degraded paint easily broke the edge detection thresholds, and the Hough Transform frequently misidentified the edges of cones, barrels, or grass boundaries as lane lines. We pivoted to a Machine Learning approach because convolutional neural networks learn semantic context, allowing them to accurately segment drivable paths and lanes even when the physical lines are partially obscured or subjected to harsh lighting gradients.
    
    \item \textbf{Model Deep Dive (\texttt{LFD\_Roadseg}):} For our core lane detection, we utilized the \texttt{LFD\_Roadseg} architecture, which employs a dual-stream design to balance high-speed inference with accurate spatial resolution. As shown in Figure \ref{fig:lfd_model}, the model processes two inputs simultaneously: a high-resolution image ($H \times W$) to preserve fine, pixel-level details like thin lane boundaries, and an aggressively downsampled low-resolution image to quickly capture the broader, global context of the road. 
    
    \begin{figure}[h]
        \centering
        \includegraphics[width=1.0\textwidth]{pipeline.png}
        \caption{Architecture overview of the LFD\_Roadseg neural network, detailing the dual-stream feature extraction and Selective Fusion module.}
        \label{fig:lfd_model}
    \end{figure}
    
    Both streams pass through independent feature extractors. The low-resolution contextual features ($F_2^l$) are processed by an Aggregation Module and upsampled ($F_a^l$) to match the high-resolution dimensions. The core of the network is the \textbf{Selective Fusion} module. It concatenates the high-res and upsampled low-res features, passing them through convolutional blocks and a Sigmoid function to generate a spatial attention mask. This mask is element-wise multiplied with the high-resolution features ($F_1^h$) to aggressively suppress background noise and highlight the drivable path, followed by a residual addition to preserve structural integrity. 
    
    Because this classifier outputs a single binary mask rather than distinct lane instances, we engineered a post-processing step. We apply a tensor squeezing operation to isolate the target dimensions before running a sliding-window histogram algorithm. This spatial histogram calculates pixel density peaks along the horizontal axis to track individual lane lines. When encountering intersections where lanes flare out and standard polynomial regression fails, the system triggers a custom radial fitting algorithm. Finally, we query the ZED2's synchronized spatial point cloud at those 2D coordinates to extract the third geometric dimension, naturally elevating our 2D classification into a 3D trajectory.

    \item \textbf{TensorRT Graph Optimization:} With the model architecture established, the primary bottleneck became inference latency. Running raw PyTorch models on the Jetson was completely unfeasible for real-time control loops. To achieve true deterministic speed, we compiled our trained \texttt{LFD\_Roadseg} model into a highly optimized TensorRT engine. TensorRT executed aggressive graph optimizations, fusing redundant layers, eliminating unused memory allocations, and quantizing our model weights from standard FP32 precision down to FP16. This unlocked massive parallel execution on the Jetson’s Tensor Cores, dropping our inference latency significantly.
    
    \item \textbf{VRAM Pipeline and Serialization Zero-Copy:} Moving to TensorRT introduced integration challenges regarding data serialization within our Python stack. TensorRT engines demand raw, contiguous memory buffers mapped precisely to the GPU's hardware layout. To bridge this gap, we engineered a high-performance Python inference wrapper that keeps the entire data pipeline resident in video memory (VRAM). Instead of allowing the CPU to slowly serialize data back and forth between host and device memory, our pipeline maps the incoming ZED2 camera buffers directly into CUDA graphics memory, runs the TensorRT execution context in-place, and processes the squeezed tensors completely within the VRAM, bypassing the Python Global Interpreter Lock (GIL) entirely.
    
    \item \textbf{Environment Optimization via Pixi:} Our final deployment challenge was environment reproducibility. We originally used Docker, but the container overhead, massive image sizes, and clunky GPU passthrough configurations created immense friction when deploying directly on the vehicle’s Jetson platform. We overhauled our environment by porting the system to Pixi. Pixi allowed us to explicitly lock our Python dependencies, CUDA toolkits, and hardware libraries in a single, lightweight Conda-based environment file, granting us native-speed execution and slashing our compilation times down to seconds.
\end{itemize}

\subsubsection*{Personal Reflection \& Future Outlook}
\begin{itemize}
    \item \textbf{Theory vs. Physical Hardware Constraints:} Looking back at this semester, the biggest reality check for me was realizing just how brutal deploying to physical hardware actually is. It’s one thing to get a model running smoothly in a Jupyter Notebook on my laptop, but getting it to survive the compute constraints of the Jetson while keeping the frame rate high enough to actually steer the car was a massive headache. The math has to actually translate to the metal. Building the model was only half the battle; fighting through serialization bottlenecks and memory management to make it fast took just as much engineering.
    
    \item \textbf{Algorithmic Improvements:} For next semester, I honestly want to try finding a model that natively splits the lanes within the road for us right out of the box. Doing all this tensor squeezing and sliding-window histogram math as a post-processing workaround is costing us speed, and it absolutely falls apart at intersections. We need to find or train something more robust to handle those flaring lane lines without relying on hacky radial fitting fail-safes. 
    
    \item \textbf{Hardware and Sensor Architecture:} Also, we are leaving a lot of efficiency on the table. Right now, we rely entirely on the ZED2's internal IMU to calculate the EKF for spatial tracking. But the camera is bolted to the car, and the car is already computing its own EKF for the main drive system. So next semester, I want to change it to jsut rely on the car's EKF instead. If we stop doing the exact same math twice, then we could also put the ZED1(has no IMU) camera on the car to maybe get a wider FOV, helping more with tight corners.
\end{itemize}
